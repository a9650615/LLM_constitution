# 15 — Token 經濟（主對話的 context 衛生）（中文鏡像）

鏡像 en v1.3（2026-07-07）。權威版：`docs/15-token-economy.md`（英文），衝突以英文版為準。
`docs/10-dispatch.md` 管的是派工側（工人活外包、回報合約、層級順序）；
本檔管**主對話自己**該做的。額度背景：`BINDINGS.md` §Budget。

## 1. 讀檔紀律

- **讀區段，不讀全檔。** 先定位（Grep/Glob 或已知行號範圍），再帶 offset/limit
  去 Read。為了用 30 行讀 2000 行，等於燒掉 98% 的花費。
- **絕不重讀已在 context 裡的東西。** Write/Edit 成功本身就確認了檔案狀態——
  自己讀回來既是浪費，也是自驗（第四法／核心三律第 2 條說 read-back 是
  fresh agent 的工作，不是你的）。
- **開場信任記錄在案的機器事實**（`BINDINGS.md` §Machine environment——但先確認
  該檔綁定的就是你所在的機器）。重新探索機器（查 OS、找工具）每次燒 10–30 次
  工具呼叫，是歷史上最大的漏洞（`docs/00-diagnosis.md` 病灶一）；反過來，
  信了他機的事實是另一種失敗。
- 預計要讀 ≥3 個檔案全文？那是派工的領域——`docs/10-dispatch.md` §1。

## 2. 輸出紀律

- **先給答案。** 不鋪背景；第一句話就是結論。
- **不把原始資料倒進對話**——log、diff、掃描結果、檔案清單超過 30 行就落檔，
  對話只給路徑＋三行摘要。（subagent 遵守的回報合約 `docs/10` §4，對你同樣適用。）
- 改完檔**不要把內容回貼**給使用者；講改了什麼、在哪（`路徑:行號`）。
- 回覆長度跟著問題重量走：一行的問題配一段話的回答，不是配一份有標題的報告。

## 3. 工具呼叫紀律

- **獨立呼叫批次發**（同一個 block 平行）；逐次呼叫只留給真正有依賴的情況。
- 工具結果已經告訴你的事，不要再跑一個指令去確認。
- 優先用專用工具（Read/Grep/Glob）而非 shell 的 `cat`/`grep`——更便宜、
  結構化、不折騰權限。
- 路由表指到的文件與 skill **需要才載入**，本 session 已載過的不重載。

## 4. Context 變長的時候

- **每完成一個子任務就落檔**（多子任務工作皆然；這套制度自己就是這樣蓋的：
  每完成一件先落地再做下一件）。session 死掉時，檔案就是倖存的全部。
- **任何做不到便宜重做的操作之前，先寫交接檔**（`commands/handoff.md` 的
  格式：目標／已完成＋證據／進行中／未驗證存疑／失敗軌跡／下一步）。「感覺腫了」**不是**
  觸發條件——計數器（子任務數／token 數）才是。
- 先收掉目前的範圍，再接新範圍進同一個 session。

## 5. 不准省的地方

- **驗證**（`docs/10` §6）——品質底線永遠不拿去換 token。
- **讀使用者的原話**——逐句重讀他們的要求，是世界上最便宜的防 bug 手段
  （`docs/20` R2）。
- **升級 prompt 裡的失敗軌跡**（`docs/10` §5）——砍掉它保證同一個失敗付兩次錢。
