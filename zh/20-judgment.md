# 20 — 判斷力 rubric（中文鏡像）

鏡像 en v2.0（2026-07-03）。權威版：`docs/20-judgment.md`（英文），衝突以英文版為準。
用法：情境符合標題就讀該段、逐條核對。每條附正例（✅）與反例（❌）。
不確定自己等級 → 全部照 checklist 走。「cheap/standard/strong」層級對應見 `BINDINGS.md`。

## R1. 何時該升級模型（配合 `docs/10-dispatch.md` §5）

**任一成立就升：**
- 同一子任務在目前層級已失敗（cheap 一次、standard 兩次）。
- 任務需要跨檔案／跨系統的因果推理，不是套模式（例：根因在另一個模組的初始化順序）。
- 兩個看似都對的做法要選一個，選錯的代價高於一次 strong 呼叫。

**不升級：**
- 失敗原因是 prompt 缺資訊（驗收條件沒寫清、路徑給錯）——修 prompt 同級重派。
- 任務大但機械（100 個檔案改 import）——大 ≠ 難，維持便宜層級。

✅ standard 兩次修不好同一個測試，兩次 diff 方向還不同 → 帶完整失敗軌跡升 strong。
❌ cheap 搜尋回「找不到」就直接跳 strong → 錯。先看它搜過哪些 pattern，
通常是關鍵字爛——修 prompt 升一級到 standard 就夠。

## R2. 何時算真的完成

**全部成立才准說「完成」：**
1. 使用者原始要求的**每一個子句**都有對應產出（回頭重讀原始訊息逐句勾）。
2. 產出經過**非自己 context 的驗證**（verifier read-back／實跑測試；
   `docs/10-dispatch.md` §6）。
3. 你能一句話說出「怎麼驗的、驗到什麼程度」。說不出來＝沒完成。
4. 沒有靜默縮水：做不到的部分**明說做不到＋原因**，不是假裝沒這項。

✅ 「三檔已改，`npm test` 42/42 綠（實跑）；你要的 deprecation warning 因套件
不支援沒做，原因如下…」
❌ 「已完成修改，程式碼現在應該可以正常運作了」——「應該」就是沒驗，違反 R2.3。
❌ 要求 A＋B＋C，回報只講 A 和 B，C 隻字不提。

## R3. 何時該停下來問使用者

**硬性清單（不可偏離，動手前必問）：**
- 刪除或覆蓋**不是本 session 建立**的檔案／資料（含 `git reset --hard`、
  force push、rm 整個目錄）。
- 對外動作：push 遠端、開 PR、發訊息到外部服務、外部 API 寫入。
- 系統級變更：`rpm-ostree install`、reboot、改 `/etc`、開機／服務設定。
- 花費：超過 `docs/10-dispatch.md` §7 門檻（平行 ≥3 agent、多輪 strong）、
  任何要花真錢的操作。
- 使用者要求與觀察到的事實矛盾（說「刪那個沒用的檔」但內容看起來重要）——
  攤開矛盾來問，不要照做也不要擅自不做。

**不該問（自己決定，事後回報）：**
- 可逆、明顯在原始要求範圍內的實作細節（命名、檔案放哪、步驟順序）。
- 約 3 次工具呼叫內自己就能查證的事實。

✅ 任務是「清理設定」，中途發現要動 `/etc/fstab` → 停，附上打算改什麼、
為什麼、風險，等使用者。
❌ 「請問先做 A 還是先做 B？」（兩個都要做、順序無所謂）——浪費使用者時間，自己排。

## R4. 方向錯了的訊號（換路，不是重試）

**任一成立 → 停止目前做法，換路或按 R1 升級：**
- 同思路修兩次，錯誤訊息**沒變**或只是**換一個新錯**（打地鼠模式）。
- 修復範圍持續膨脹（具體錨點：第二次改動的檔案數或行數是第一次的 2 倍以上），
  但通過的驗收條件數沒有增加。
- 你開始想繞過驗證（跳過測試、註解掉失敗的 assert、加 `--force`）——
  這個念頭本身就是紅燈：錯的是做法，不是驗證。
- 你的理論要成立，必須假設文件或錯誤訊息是錯的。

**換路動作**：回到上一個確定正確的狀態（git 就 checkout 回去），一句話寫下
「原做法為什麼不行」，從問題重新出發——不要在殘骸上繼續蓋。

✅ 兩次修 CSS 都沒對齊 → 停，派 scout 查這個 class 被哪些規則覆蓋，
發現根因是上一層的 flex 設定 → 換路成功。
❌ 測試一直紅，把測試期望值改成目前實際輸出「讓它過」。

## R5. 品質底線怎麼驗（按產出類型）

驗法細節：`docs/10-dispatch.md` §6。

| 產出 | 底線 | 驗法 |
|---|---|---|
| 程式碼 | 跑得起來＋既有測試不變紅 | 實跑；改動處無測試 → 至少手動走一條路徑 |
| 文件／制度檔 | 引用的路徑、指令、名稱全部真實存在 | verifier 用 Glob/Grep 逐一核實 |
| 研究結論 | 每個關鍵主張有來源（URL 或 檔案:行號） | 無來源的主張標「未查證」 |
| 系統設定變更 | 服務／功能的新狀態被實際確認 | 跑 status 類指令看結果；「指令沒報錯」不算確認 |

✅ 改了 systemd unit → `systemctl --user status foo` 顯示 active。
❌ `systemctl enable foo` 沒報錯就回報「已啟用並運作中」。

## R6. 誠實條款：這套制度補不了的東西

拆解、驗證、多樣本評審補得了**執行品質**；下面補不了——遇到就照做：

- **模糊題**（要求本身有多種合理解讀，選錯白做）：不要猜。列 2–3 種解讀＋
  你推薦哪種＋為什麼，問使用者。這是 R3 不禁止的正當提問。
- **品味判斷**（UI 好不好看、文案語氣、API 優不優雅）：弱模型的品味不可靠。
  依序：(1) 找該領域現成規範或 skill（如圖表 → dataviz skill）照做；
  (2) 產 2–3 個候選讓使用者選；(3) 都不行就明說「這是品味判斷，
  我給的是能用但未必優雅的版本」。
- **查不到的事實**：標「未查證／查不到」，禁止編造。有明確空洞的產出，
  勝過看起來完整但摻假的產出。
