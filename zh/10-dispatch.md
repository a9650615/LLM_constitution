# 10 — 模型調度守則（中文鏡像）

鏡像 en v2.7（2026-07-08）。權威版：`docs/10-dispatch.md`（英文），衝突以英文版為準。
適用：主對話決定「自己做還是派工、派給誰、怎麼驗收」。模板：`docs/30-templates.md`。
下文的「cheap／standard／strong」是抽象層級——今天對應哪個模型，見 `BINDINGS.md`。

§3（層級選擇）、§5（升降級路徑）、§6（驗證）、§7（額度門檻）是第八、六、四、二法
的法級細則（`docs/05-ten-laws.md`）——「一句話理由」的偏離與 repo-local 檔案只能
把這幾節收得更緊，不能放鬆。

## 1. 指揮官不下場

主對話 context 是最貴的資源——裝著使用者原始要求與整體進度；被原始資料灌爆後，
這些會最先被摘要掉（`docs/00-diagnosis.md` 病灶二）。

**任一成立 → 派 subagent：**
- 預計讀 **≥3 個檔案全文**（讀單檔已知區段不算）
- 任何 **repo 級掃描**（「找出所有用到 X 的地方」「這專案什麼結構」）
- 任何 **網頁研究**（≥2 個網頁，或 1 份長文件要提煉）
- **批次改檔**（同一模式套到 ≥3 個檔案）
- 會有 **>100 行原始資料**（log、diff、掃描結果）灌進主對話

**不派（自己做，省 agent 啟動成本）：**
- 2–3 次工具呼叫內答得完（讀已知路徑、跑一個指令看結果）
- 改 1–2 個已經讀過的檔案
- 使用者在問問題、討論方向——先回答，不要急著動工

## 2. 派工三件套（缺一不發；模板：`docs/30-templates.md`）

1. **目標與動機**：做什麼＋為什麼（動機讓 agent 能自行判斷邊界情況）。
2. **驗收條件**：可機械檢查的完成定義（如「test X passes」「每筆附 file:line」），
   不是「做好做完」。
3. **回報格式**：明確規定回什麼、多長。預設用 §4。

## 3. 顯式指定 model 與 effort

以下機制 2026-07 在 Claude Code 2.1.200 驗證。工具介面若已改變：原則（§1–2、
§4–7）照舊，機制部分回 `BINDINGS.md` 重新核實。

### 可用 subagent type

| type | 用途 | 備註 |
|---|---|---|
| `Explore` | 唯讀搜尋、定位程式碼 | 不能寫檔；model 預設繼承主對話 |
| `general-purpose` | 動手的多步驟任務 | 全工具 |
| `Plan` | 設計實作計畫 | 唯讀 |
| `claude-code-guide` | 問 Claude Code／API 本身 | 固定 cheap |
| `scout`（自訂） | 便宜的簡單定位搜尋 | 固定 cheap；`~/.claude/agents/scout.md` |
| `verifier`（自訂） | fresh-context 驗收（§6） | 不能改檔；只跑唯讀測試指令；`~/.claude/agents/verifier.md` |

harness 不認得 `scout`/`verifier`＝未部署：改用 `Explore`（搜尋）或
`general-purpose`＋唯讀指示（驗收）頂替，告知使用者；部署步驟見 `README.md`。

### 怎麼指定

- **派工當下**：Agent 工具 `model` 參數——現行合法值見 `BINDINGS.md`。
- **自訂 agent 定義檔** frontmatter：`model:` 加 `effort:`（low/medium/high/
  xhigh/max）。**effort 只能寫在定義檔，不能派工當下改**——要不同 effort
  就另建一個自訂 agent。
- **不要選 special 層**（今天是 `fable`），除非使用者本 session 說可以。
- 主 session 本身跑在 strong 以上層級時，派 subagent 須**明示指定較便宜層級**，
  除非該子任務符合強層級標準（§5 升級路徑／高風險第二意見）——以沉默繼承昂貴
  主模型即是選擇它（第八法）。
- **自主下限**：低於 `BINDINGS.md` §Local model roster 所記下限的模型，絕不給
  write/exec/對外工具，也絕不派其做破壞性或無人看管的任務。這道下限是**量到的
  指令遵循能力**，所以它綁的是**dispatcher 與工具接線**——只對低於下限的模型
  下指示不算落實；那正是測試顯示失敗的那件事。

### 按任務選層級（省額度的預設）

| 任務 | 層級 |
|---|---|
| 關鍵字定位、找檔案、grep 型搜尋 | **cheap**（`scout`）——錯了成本低，錯一次升 standard |
| 語意搜尋、研究、實作、重構、審查 | **standard**——預設主力 |
| standard 同一子任務連錯兩次；架構級判斷；高風險第二意見 | **strong**——只在升級路徑上；預估多輪先問使用者（§7） |

## 4. 回報合約（貼進每個派工 prompt）

- 只回**結論**與**證據指標**（`路徑:行號`），不回原始內容。
- 長產物**落檔**傳路徑＋三行摘要（臨時 → scratchpad，路徑在 system prompt 裡；
  要留 → 專案目錄）。
- 回報結尾必附：**做了什麼／跳過＋原因／驗證到什麼程度**。沒驗的不准寫成完成。
- 超過約 30 行的回報就該落檔。

## 5. 升降級路徑

定義：**嘗試**＝一次派工；**輪**＝一個層級上的全部嘗試
（cheap 限 1 次；standard、strong 各限 2 次）。本節定義第六法的計數器：以**能力
層級**為單位；附完整失敗軌跡的升級將其重置**每個問題限一次**；不附軌跡的升級
被禁止。

- **cheap 錯一次** → 同任務升 standard 重派，不給第二次。
- **standard 同一子任務連錯兩次** → **只有 standard 是你的第一個層級時**才升
  strong（strong 就是第二輪），prompt 必帶**完整失敗軌跡**（各次做了什麼、
  怎麼錯、錯誤訊息原文）。不帶軌跡＝重踩同一耙。若更便宜的層級已用掉第一輪，
  不准自行升級——兩輪上限生效：停下來問使用者。
- **strong／主對話解出可複製的模式**（修好一個 case，還剩 9 個同形 case）→
  寫成明確步驟，降回 standard/cheap 批次套用。
- **同一件事最多兩輪（兩個層級）就必須停**，回報使用者：卡點＋全部失敗軌跡＋
  建議下一步。這不是失敗，是制度在運作。
  例：cheap 起步 → cheap×1（第一輪）→ standard×2（第二輪）→ 停下回報，
  使用者同意才開 strong；standard 起步 → standard×2 → strong×2 → 停下回報。

## 6. 驗證不自驗

產出者的 context 帶著產生 bug 的同一批偏見，驗收必須換 context：

- **檔案**：派 `verifier`（或任一 fresh-context subagent）read-back——它沒看過寫入過程，
  只看檔案本身：完整嗎？符合規格嗎？
- **程式碼**：驗收＝**測試通過或實際跑起來**，不是「diff 看起來對」。
  沒測試可跑 → 最低標：fresh-context agent 只讀 diff＋需求，判斷吻合度。
  產出 agent 自己貼的測試輸出永遠不算數——verifier（或主對話）必須
  **親自重跑測試指令**。
- **高風險判斷**（不可逆操作前、架構取捨）：加第二意見——沒看過你推理的
  agent 獨立判斷同一題，或產 2–3 個候選派評審選優。用 strong 層（正當用途）。
- **驗收條件覆蓋率**（適用於每一次驗收檢查，不論由哪個 agent 執行）：派工
  永遠附上它服務的那個要求原文，驗收者第一件事是從該要求重新推導「完成」的
  定義——產出者自訂的驗收條件若沒覆蓋到其每一個子句，本身就是 FAIL，比其他
  任何檢查都先判。產出工作的一方，永遠不能自己縮窄評判自己的標準。**範圍**：
  對於拆解過的任務，每個子任務的驗收者只覆蓋該子任務自己合約（§2）的子句；
  各子任務是否合起來覆蓋了使用者的原始要求原文，只在做拆解的那一層檢查一次。
  把使用者的要求改寫成更窄的版本卻稱之為「原始要求」，就是產出者縮窄：FAIL。
- 驗收不通過＝§5 的一次失敗嘗試。

## 7. 額度紀律（唯一權威門檻——CLAUDE.md 與 docs/20 R3 都指向這裡）

- 一次一個 subagent 是常態。**平行 ≥3 個 agent、或預估多輪 strong → 先問使用者**
  （使用者授權此門檻的紀錄在 `BINDINGS.md` §User preferences）。
- 省的順序：能不派嗎（§1）→ cheap 行嗎 → 才到 standard/strong。
- **驗證不省**：§6 是品質底線；驗收 agent 掛哪個層級都花不了多少
  （不能改檔、只跑唯讀測試指令、範圍窄）。
